HTML —
Structure & Semantics

HyperText Markup Language (HTML) is the skeleton of every webpage. It uses elements — written as tags — to describe the meaning and structure of content. Semantic HTML (using <header>, <nav>, <main>, <article>, <footer> etc.) improves accessibility, SEO, and code readability. Introduced in 1993, HTML5 — the current standard — added native support for media, forms, and semantic landmarks.

Demonstration: Semantic Document Structure

The code below shows a semantically correct HTML page skeleton. Each tag communicates meaning — not just appearance — to browsers and screen readers. A non-semantic version would use <div> everywhere, which conveys nothing to assistive technology.

<!-- Semantic page structure --> <header> <nav> <a href="#home">Home</a> <a href="#about">About</a> </nav> </header> <main> <article> <h1>Article Title</h1> <p>Article body content goes here.</p> <figure> <img src="photo.jpg" alt="Descriptive alt text"> <figcaption>Caption for the image</figcaption> </figure> </article> <aside>Related links or sidebar content</aside> </main> <footer> <address>Contact: <a href="mailto:me@example.com">me@example.com</a></address> </footer>
Why this matters: Screen readers use semantic tags to navigate. Search engines rank pages partly on semantic correctness. <article> tells a crawler this is self-contained content; <aside> signals supplementary material. Using <button> instead of <div onclick> gives keyboard and screen-reader users full interaction support for free.

Reference

Essential Elements

HTML5 defines over 100 elements. These are the ones you'll use on almost every project — organised by their semantic role.

html5 element reference
<header>

Introductory content or a group of navigational aids. Usually contains a logo and the primary nav.

<nav>

A section of a page whose purpose is to provide navigation links, either within the document or to other pages.

<main>

The dominant content of the document. Only one <main> per page; it excludes sidebars and navs.

<article>

Self-contained content that could be independently distributed — a blog post, comment, or news story.

<section>

A thematic grouping of content with its own heading. Use when content warrants its own navigation entry.

<aside>

Content tangentially related to the main content — pull quotes, sidebars, related links, or ads.

<figure>

Self-contained content like images, diagrams, or code listings, optionally with a <figcaption>.

<footer>

Footer of its nearest sectioning element. Contains authorship info, copyright, or related document links.

<button>

A clickable control. Preferred over <div onclick> for keyboard and screen-reader accessibility.

<details> / <summary>

Disclosure widget — a native accordion/spoiler with no JavaScript needed.

<time>

Represents a specific period in time. The datetime attribute provides machine-readable format.

<picture>

Responsive images — serve different image files based on screen size or display density via <source>.

Accessibility rule of thumb: If an element is interactive, use a native interactive element (<button>, <a>, <input>). Native elements are focusable, announce their role to screen readers, and respond to keyboard events — all for free. Recreating this with <div> requires setting role, tabindex, and custom keyboard handlers.
Deep Dive

HTML5 Form Validation

HTML5 introduced built-in form validation that works without any JavaScript. Constraint validation attributes like required, minlength, pattern, and type are evaluated by the browser before submission.

demo · html5 constraint validation
<!-- HTML5 validation — zero JavaScript needed --> <form> <!-- required: must not be empty --> <input type="text" name="name" required> <!-- type="email": browser validates format --> <input type="email" name="email" required> <!-- pattern: regex must match --> <input type="text" pattern="[A-Z]{3}[0-9]{4}" title="3 uppercase letters + 4 digits"> <!-- min / max: numeric range --> <input type="number" min="18" max="120" name="age"> <!-- minlength / maxlength: character count --> <input type="password" minlength="8" name="pwd"> <button type="submit">Submit</button> </form>
Client-side vs server-side validation: HTML5 constraint validation catches obvious errors before the network round-trip, improving UX. However, it is never a security measure — a determined user can bypass it with browser developer tools. Always re-validate on the server.
